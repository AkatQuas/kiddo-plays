AI每日代码Commit分析推送系统 PRD

1.  产品概述
    1.1 需求背景
    团队日常Git代码协作过程中，多仓库代码提交分散、提交记录零散无序，缺乏统一的每日迭代汇总机制。传统人工复盘模式存在成本高、效率低、遗漏量大、无法量化团队迭代进度、难以识别代码变更风险等问题，管理人员及研发成员无法快速、精准知晓当日各仓库代码变更范围、变更类型、潜在风险、迭代进度与个人工作产出。
    基于此，亟需一套全自动、轻量零运维、本地闭环、高稳定、可追溯的代码日报自动化系统，无需人工介入，自动抓取多仓库每日增量Commit，通过AI智能拆解、分类、分析代码变更，输出结构化、可读化的团队迭代日报，并统一推送至内部协作渠道，实现团队开发进度、代码质量、变更风险的透明化同步。
    1.2 适用场景与用户角色
    适用场景：研发团队日常迭代复盘、每日代码变更汇总、迭代风险巡检、团队工作产出量化、多仓库统一管理同步
    用户角色：研发工程师、技术负责人、项目管理人员
    1.3 方案选型结论
    采用Crontab定时调度 + Python核心脚本 + 外置配置文件 + Claude Code CLI本地AI分析 + 通用Webhook推送的轻量落地方案，全程本地服务闭环，无云端依赖、无常驻进程、零运维成本。
    各组件核心作用：

        - Crontab：实现服务端定时自动触发，适配每日固定迭代汇总场景
        - Python脚本：核心调度、仓库管理、增量采集、并发控制、日志汇总、消息推送、异常容错
        - 外置配置文件：解耦业务逻辑与配置，支持零代码修改参数、新增/删除仓库
        - Claude Code CLI：本地离线AI智能分析，拆解代码变更、分类工作内容、生成迭代总结
        - Webhook推送：适配飞书/钉钉/企业微信多渠道消息推送，统一团队同步口径

    1.4 核心目标

        - 全自动无人值守：每日定时自动执行，全程无需人工干预，执行完毕自动退出，无资源常驻
        - AI结构化解析：将零散、原始的Git Commit记录，转化为按人员、按类型分类的标准化团队日报，可读性极强
        - 风险可视化识别：自动区分新增功能、Bug修复、代码优化、重构变更，精准识别潜在高危代码调整
        - 统一信息同步：单渠道统一推送全仓库汇总日报，杜绝信息碎片化、同步不及时问题
        - 高稳定可追溯：全流程日志留存、断点续扫、异常隔离，支持问题溯源与手动兜底修复
        - 轻量易维护：低资源占用、配置化管理、无硬编码，支持快速部署、迁移、迭代扩展

2.  整体业务流程

    全局前置校验 → 读取全局+仓库配置文件 → 初始化本地目录/日志环境 → 扫描并归档冗余废弃仓库 → 多仓库并行任务调度（带并发限流） → 单仓库增量Commit采集与预处理 → 调用Claude CLI AI智能分析 → 等待全量仓库任务执行完毕 → 汇总正常结果+异常信息+空数据场景 → 生成标准化团队日报 → Webhook一次性全局推送 → 批量更新成功仓库游标文件 → 全流程日志落地归档 → 任务自动退出
    核心特性：纯单次瞬时任务、无前端交互、执行即退出、幂等运行、单仓库故障隔离、全局不中断、全场景兜底、数据可追溯

3.  详细功能需求
    3.1 定时调度能力

        - 调度方式：服务端Crontab系统级定时任务，稳定可靠，无程序内调度故障风险
        - 默认执行时间：每日23:00自动执行，汇总当日全天代码迭代内容
        - 配置化自定义调度：支持通过全局配置文件修改执行时间、关闭定时任务，无需改动核心业务代码
        - 任务幂等性保障：基于仓库游标实现增量幂等执行，重复触发任务不会重复统计已分析Commit；兼容首次初始化批量采集、手动重试执行、异常重试场景
        - 手动触发能力：支持脚本手动单次执行，适配临时复盘、异常补跑、功能调试场景，手动执行可自定义统计周期

    3.2 仓库代码拉取与Commit采集

    3.2.1 配置化多仓库管理
    核心逻辑完全解耦仓库配置，无任何硬编码仓库信息，通过独立外置仓库配置文件实现批量管理，支持无限新增、编辑、删除仓库，无需修改脚本核心逻辑。单仓库支持独立配置：SSH仓库URL、监听分支、本地独立工作目录、仓库展示名称、是否启用分析开关。
    新增仓库启停开关：支持临时关闭某仓库分析任务，无需删除配置，适配仓库迭代暂停、维护场景。

    3.2.2 仓库自动初始化与更新
    遍历配置文件中所有已启用合规仓库，基于全局仓库根目录拼接本地工作目录，自动校验并维护本地仓库完整性：- 本地无对应仓库目录：通过配置的SSH地址自动克隆代码至指定目录，适配私有仓库SSH密钥授权模式 - 本地已存在仓库目录：进入对应目录，拉取指定监听分支最新代码，保证本地代码与远程分支完全同步
    私有仓库适配补充：兼容服务器本地SSH全局密钥授权，无需脚本配置密钥，保障私有仓库安全拉取，无密钥泄露风险。

    3.2.3 冗余仓库安全清理机制
    脚本启动优先执行冗余仓库巡检，对比本地仓库目录与最新配置文件仓库列表，精准识别废弃冗余仓库：本地存在仓库目录/游标文件，但未出现在当前有效配置列表中，判定为废弃资源。
    所有废弃仓库目录、对应游标文件统一迁移至系统 /tmp 临时目录归档，不直接物理删除，避免误删有效数据；系统自动清理tmp过期文件，同时脚本记录完整归档日志（仓库名称、原路径、迁移时间、操作状态），支持手动溯源恢复，兼顾整洁性与数据安全性。合规仓库数据零修改、零删除。

    3.2.4 增量游标断点续扫机制
    采用单仓库独立Commit游标模式，每个仓库对应独立游标文件，精准记录上一次成功分析的最新有效Commit ID，实现增量采集。- 游标初始化规则：仓库首次执行、无游标文件时，不默认当日零点截取，自动采集最近7天内所有有效Commit完成初始化分析；执行成功后写入最新Commit游标，后续全程增量续扫 - 游标更新规则：单仓库任务完全执行成功（代码拉取+AI分析无异常），自动覆盖更新最新Commit游标；任务失败、超时、异常中断时，不更新游标，下次任务从原游标位置重试，杜绝数据遗漏 - 游标读写锁机制：新增文件读写锁，防止多仓库并行任务并发读写游标文件，导致游标数据错乱、覆盖丢失问题 - 手动兜底规则：支持手动删除/修改游标文件，自主重置采集起点，适配初始化异常、数据错乱、历史补扫场景

    3.2.5 Commit过滤规则
    全局自动过滤无效提交，仅保留研发有效开发记录，规避冗余数据干扰日报质量：- 强制过滤所有Merge合并提交 - 自动过滤空提交、无代码变更提交、仅注释修改无逻辑变更的无效提交 - 过滤Git自动生成的模板提交、回滚空记录

    3.2.6 多仓库并行调度机制
    合规仓库确认后，采用多进程并行执行单仓库任务，各仓库拉取、采集、AI分析流程互不阻塞。新增全局并行限流配置，可自定义最大并发任务数，防止同时启动大量Claude进程、Git进程导致服务器CPU/内存资源过载，保障服务稳定。
    脚本仅向AI传递仓库本地路径+增量Commit短ID列表，无任何代码预处理逻辑，所有文件变更、代码逻辑、改动细节由AI自主只读分析，轻量化脚本运行压力。

    3.3 通用CLI Coding Agent AI分析核心能力
    系统采用AI引擎抽象层 + 多Agent适配器插件化架构，彻底解除业务逻辑与单一Claude Code的硬绑定，原生支持横向扩展接入主流CLI Coding Agent工具（Claude Code、GitHub Copilot CLI、豆包Code CLI、通义灵码CLI等），无需改造核心调度、采集、推送逻辑。当前默认适配Claude Code CLI，所有Agent统一遵循「单次非交互执行、执行完毕自动退出、只读安全约束、标准化结构化输出」规范，完全适配脚本自动化运行场景，杜绝任务卡死、进程残留、服务失控问题。
    架构设计核心：插件化适配器
    核心调度层不感知具体AI引擎，仅调用统一抽象接口；各CLI Coding Agent通过独立适配器适配参数、调用命令、安全约束、输出解析，新增任意AI引擎仅需新增对应适配器，零侵入核心代码，完美保留系统扩展性。

    3.3.1 通用AI引擎全局约束 - 统一运行模式：所有CLI Coding Agent强制使用单次查询非交互模式，无对话上下文留存、无交互式等待，执行完成立即退出进程，适配自动化脚本调度 - 统一安全只读约束：所有AI引擎强制开启工具白名单限制，仅允许文件读取、检索、文本查看类只读操作，禁止任何代码增删改、文件写入、命令执行、递归调用等高危操作，杜绝AI失控、服务器资源篡改风险 - 统一轮次与超时兜底：全局统一可配置AI思考最大轮次、单仓库分析超时阈值，任意引擎超时/卡死自动终止进程、标记异常、记录日志，不阻塞全局任务，保证任务闭环可控 - 统一输出协议：所有接入的AI引擎必须输出统一结构、统一分类、统一格式的日报内容，上层汇总、推送逻辑无需适配修改

    3.3.3 AI提交强制四分类规则
    - 功能新增：新接口、新页面、新业务逻辑、新功能模块、新配置能力开发，可对外提供新增业务能力的变更
    - 问题修复：线上/测试Bug修复、代码异常捕获优化、报错修复、兼容性适配、数据异常修复、逻辑漏洞修复
    - 性能/逻辑优化：代码执行效率提升、逻辑简化、参数优化、资源占用优化、容错逻辑增强、无新功能无Bug修复的提质变更
    - 代码重构：代码结构调整、模块解耦、命名规范统一、注释补充、代码格式化、冗余代码删减，无任何业务逻辑变更

    3.3.4 AI输出标准化规范

        - 人员维度聚合：摒弃单Commit逐条展示模式，自动聚合同一开发者当日所有有效提交，以研发人员为核心维度汇总产出
        - 类型维度归类：每位开发者的工作内容严格按照四大分类归类展示，无零散、无序记录
        - 自主深度分析：AI自主读取Commit对应文件变更、代码逻辑、业务改动，拆解核心工作价值，无需脚本预处理解析
        - 全局迭代总结：每个仓库独立生成小结，最终输出全仓库整体迭代总结，包含开发重心、迭代方向、代码质量、稳定性、潜在风险、进度完成情况
        - 格式适配规范：纯文本简洁输出、层级清晰、无Markdown复杂格式、无冗余话术，完美适配各类协作群消息展示

    3.4 统一汇总推送能力
    3.4.1 汇总推送机制
    摒弃单仓库分散推送模式，待所有仓库并行任务全部执行完毕后，统一汇总：正常仓库分析结果、异常失败仓库提示、无更新仓库提示，整合为一篇完整团队日报，全局仅推送一次，保证信息统一、无刷屏干扰。

    3.4.2 推送核心架构
    推送模块遵循依赖倒置原则：高层业务调度逻辑依赖「统一抽象接口」，不依赖任何具体推送渠道实现，彻底解除核心代码与飞书/钉钉/企业微信等渠道的硬编码耦合。
    所有具体渠道实现完全外置，通过配置驱动动态挂载，实现控制反转：传统代码主动绑定渠道逻辑，改为配置被动注入实现，新增、启停、修改推送渠道无需改动核心调度、日报汇总、任务流转逻辑，完全满足插拔式扩展。
    轻量IOC架构分层（无容器、纯设计范式）- 高层抽象层（稳定不变）：定义全局统一推送抽象接口、统一入参、统一出参、统一重试/异常规范，是所有推送实现的唯一依赖，核心业务只调用抽象、不感知实现 - 具体实现层（可动态扩展）：各推送渠道独立适配器，实现抽象接口规范，封装各渠道差异化协议、请求格式、签名规则，可自由增删，- 配置驱动层（控制反转入口）：通过外置配置声明「当前需要启用的推送适配器」，由配置决定程序运行时加载哪些实现，而非代码硬编码决定，实现控制权由代码内部转移至外部配置 - 业务调度层：统一调用抽象接口方法，完全与底层渠道实现解耦，自动适配配置中启用的所有推送渠道
    推送模块采用统一推送抽象接口 + IOC容器注入 + 渠道适配器插件化设计，完全解除业务逻辑与具体推送渠道的硬编码耦合。核心调度层只依赖顶层抽象推送接口，不感知飞书、钉钉、企业微信等具体实现，所有渠道能力通过适配器注入IOC容器，实现新增渠道零改核心代码、配置化动态启用/禁用，满足长期扩展性设计。
    架构分层说明 - 顶层抽象接口（统一推送标准）：定义全局统一推送方法、入参结构、返回结构、异常规范、重试机制，所有渠道适配器必须实现该标准接口 - 配置驱动IOC核心逻辑：摒弃容器实例化管理，程序启动读取外置配置，根据配置声明动态加载对应适配器模块，完成抽象与实现的动态绑定，实现控制反转 - 渠道适配器（具体实现）：各推送渠道独立适配层，封装对应渠道的请求头、签名、消息格式、协议规则，互不侵入 - 业务调度层：只调用IOC容器统一推送入口，无需适配各类渠道差异化协议

    3.4.3 内置渠道适配器能力（IOC插拔式扩展）- 全局通用重试策略：所有注入的推送渠道统一复用重试机制，默认失败自动重试2次、重试间隔3s，支持配置化修改重试次数与间隔 - 单渠道故障隔离：多渠道组合推送场景下，某一个渠道推送失败、超时、网络异常时，仅记录当前渠道错误日志，不影响其他渠道推送，不中断全局任务流程 - 推送状态可追溯：每个渠道推送结果、重试次数、异常信息独立落地日志，支持精准排查渠道推送问题

    3.4.4 统一重试与容错机制（全局适配器通用）
    系统默认内置主流协作渠道适配器，全部通过IOC配置动态加载，支持单选、多选、组合推送：- 飞书机器人适配器 - 钉钉机器人适配器 - 企业微信机器人适配器 - 通用HTTP Webhook适配器（适配自研/第三方推送服务）
    扩展规则：新增自定义推送渠道（短信、邮件、自有OA、消息中台等），仅需新增对应适配器实现标准接口，在IOC配置中注册即可，无需修改日报生成、汇总、调度核心逻辑。
    3.4.5 日报固定消息结构

头部标题+统计周期+全仓库汇总概览（仓库总数/更新仓库数/异常仓库数）+ 分仓库详细开发者贡献明细 + 单仓库迭代小结 + 全局整体迭代总结 + 异常仓库备注（如有）+ 空数据兜底提示（如有）

3.4.6 全场景空数据兜底
所有仓库本轮无任何新增有效Commit时，自动推送标准化「当日全仓库无新增代码迭代更新」提示，页面无空白、无报错，正常批量更新所有仓库游标，保证次日任务正常续扫。

3.5 日志与异常容错能力（全面补全）
3.5.1 全流程日志规范（新增分级+过期清理）- 日志存储位置：统一存放于配置文件同级log目录，目录自动初始化，无需手动创建 - 推送渠道配置注册（轻量IOC、配置驱动插拔，无容器依赖）- 日志分级：区分INFO正常日志、WARN警告日志、ERROR错误日志，方便快速排查问题 - 日志内容全覆盖：任务启停时间、并发配置、仓库巡检结果、冗余归档记录、Commit采集列表、AI分析详情、汇总内容、推送状态、所有异常报错堆栈信息 - 日志过期清理：支持配置日志最大留存天数（默认30天），自动清理过期日志，防止磁盘空间占用过载

3.5.2 全场景异常隔离与兜底 - 无增量Commit场景：单仓库提示「本轮无新增代码更新」，正常更新游标，任务正常闭环，无报错 - Git拉取失败：网络异常、权限不足、分支不存在等场景，单仓库标记异常、记录详细日志，不更新游标，不影响其他仓库任务 - AI分析超时/失败：超时、AI调用异常、参数错误等场景，终止当前仓库任务，标记异常，不更新游标，日报统一标注异常提示 - 网络波动/脚本异常：单仓库故障完全隔离，全局任务不中断，保留原有游标断点，下次重试续扫，杜绝数据遗漏 - 磁盘空间不足：检测到磁盘空间过低时，终止任务，记录告警日志，防止日志/文件写入失败 - 配置文件解析异常：配置格式错误、参数缺失时，全局终止任务，输出明确配置错误日志，提示用户修复配置 - Crontab重复触发：基于任务执行状态锁，防止同一时间多进程并行执行任务，避免重复采集、重复推送

4.  配置项设计
    全程采用配置与逻辑完全分离架构，所有可变参数外置配置，无硬编码，分为全局基础配置、多仓库独立配置两大类，游标文件系统自动生成维护。
    4.1 全局基础配置

        - 推送渠道注册配置（支持多渠道适配器）
        - 仓库全局根目录（所有仓库统一克隆存储路径）
        - 游标文件存储目录
        - AI分析引擎类型配置（支持切换多CLI Agent，默认claude-code）
        - 全局最大并行任务数（限流控制）
        - AI单仓库分析超时时间
        - Webhook推送重试次数、重试间隔
        - 日志最大留存天数，默认 30 天
        - 定时任务执行时间（Crontab表达式配置）
        - 初始化默认采集回溯天数（默认7天，可配置）

    4.2 单仓库独立配置 - 仓库SSH远程URL（私有仓库适配）- 目标监听分支（main/master/dev等自定义分支）- 仓库自定义展示名称（日报展示用）- 仓库启用开关（true/false，临时停用，分析时跳过，不必删除仓库内容）
    4.3 游标文件规则
    系统采用追加式游标日志机制，摒弃传统单值覆盖存储模式，为每个已启用仓库独立生成游标日志文件，全程Append追加写入、不覆盖历史记录，实现游标可追溯、可回滚、可复盘。
    游标文件命名规则：仓库名\_分支.cursor.log
    存储结构（每行一条有效执行记录）：每条日志固定存储「执行时间戳｜本轮最终Latest Commit ID」，格式标准化、机器可读、人工可自查。
    游标读取逻辑：系统每次启动增量采集时，自动读取当前游标日志文件最后一行有效成功记录，提取其中Commit ID作为本次增量分析的起始游标，精准实现断点续扫。
    游标写入规则：仅当单仓库全流程执行成功（代码拉取+AI分析+无异常）后，才向游标日志文件Append追加一条新记录；任务失败、超时、异常中断时不写入、不追加，保留上一条有效游标，下次继续重试续扫，杜绝数据遗漏。
    初始化规则：仓库首次执行、无游标日志文件时，自动初始化采集最近7天Commit，并写入第一条游标记录。
    手动运维兜底：支持手动删除单行记录、清空日志、回滚至任意历史Commit节点，灵活适配数据错乱、异常补扫、历史复盘场景。

5.  非功能需求
    - 轻量低耗：无常驻进程，仅每日瞬时执行，CPU、内存占用极低，适配低配服务器运行
    - 高稳定性：链路短、依赖少、故障点隔离，单节点故障不影响全局，支持长期7\*24小时自动化运行
    - 易维护可迁移：配置与逻辑解耦，代码结构清晰，支持一键部署、快速迁移、批量扩容仓库
    - 高安全性：无密钥硬编码、无公开暴露接口、全程本地闭环运行、配置文件权限加密、游标文件防篡改、无高危操作
    - 可观测可追溯：全流程日志留存、异常精准定位、操作全程记录，支持问题复盘与溯源
    - 兼容性强：适配主流Linux服务器、Python3.8+、Git2.0+、Claude Code最新稳定版本

6.  安全专项规范
    - 文件权限安全：配置文件、游标文件、日志文件默认设置最小权限，禁止公共用户读写，防止配置篡改、数据泄露
    - AI行为安全：严格只读工具白名单，禁止任何文件修改、命令写入、递归调用，杜绝AI操作服务器资源
    - 仓库数据安全：冗余仓库仅迁移不删除，合规仓库数据零修改、零删除
    - 网络安全：仅主动发起Webhook推送请求，无被动监听端口，无外网暴露风险，本地闭环运行
    - 授权安全：依赖服务器本地SSH密钥授权，脚本不存储任何仓库账号密码、密钥信息，无 credential 泄露风险
    - 推送适配器插拔扩展配置文件（轻量IOC设计，无容器实现）

7.  边界规则与兜底逻辑
    整合全文所有边界场景，统一标准化兜底规则，保证任务100%闭环无报错、无数据遗漏：
    - 游标区间无Commit：正常推送空更新提示，更新游标，任务正常结束
    - 单仓库Git拉取失败：隔离异常、记录日志、不更新游标、日报标注异常
    - AI分析超时/失败：终止任务、记录报错、不更新游标、全局不中断
    - 推送失败：自动重试，重试失败记录日志，任务正常闭环
    - 配置文件异常：全局终止，输出明确错误提示
    - 冗余仓库清理：仅迁移废弃资源，合规数据零改动，全程日志溯源
    - 并发读写冲突：文件锁机制规避游标文件错乱问题
    - 手动重置游标：支持任意节点重启增量采集，适配异常修复
