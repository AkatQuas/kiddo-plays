# 文件上传下载管理系统 PRD

## 1. 文档概述

### 1.1 需求背景

搭建一套轻量、可快速部署的文件托管管理系统，基于 Docker Compose 统一编排服务，通过 Kong 网关统一路由分发，实现**用户维度的文件上传、文件元数据存储、用户文件列表查询、直连文件下载**完整闭环能力。系统区分文件二进制数据与业务元数据存储，MinIO 承载文件OSS存储，MongoDB 承载业务关联数据存储，FastAPI 提供核心业务接口与静态页面服务。

### 1.2 整体目标

- 基于 Docker Compose 实现一键部署，服务解耦、独立可维护

- 实现用户模拟维度的文件绑定管理（无需真实登录，用户名作为唯一关联维度）

- 实现前端直传 MinIO、直下 MinIO，减轻后端服务 IO 压力

- 后端仅负责元数据存储、关联关系管理、列表查询，各司其职

- Kong 统一入口，实现路由分发、统一网关管理

### 1.3 运行环境

本地/服务器 Docker + Docker Compose，所有服务容器化部署，无外部依赖，支持单机快速部署测试。

## 2. 系统整体架构

### 2.1 服务组件清单（Docker Compose 编排）

整体四大核心服务，独立容器部署，Kong 作为统一流量入口：

1. **Kong 网关**：统一流量入口、路由分发，转发前端静态页面请求、后端接口请求

2. **FastAPI Python 服务**：核心业务服务，提供静态页面访问、元数据存储接口、文件列表查询接口、健康检查接口

3. **MongoDB**：持久化存储「用户-文件」关联元数据，不存储文件二进制数据

4. **MinIO OSS**：对象存储，存储用户上传的所有文件二进制数据，提供原生 S3 上传/下载能力

### 2.2 架构分层逻辑

- **展示层**：FastAPI 托管静态 HTML 页面（上传页、下载页）

- **网关层**：Kong 统一路由，区分页面请求、接口请求

- **业务层**：FastAPI 处理业务逻辑、元数据读写、关联关系管理

- **数据存储层**：MongoDB（业务元数据）、MinIO（文件二进制数据）

### 2.3 核心设计理念

- **读写分离、职责拆分**：MinIO 只存文件、MongoDB 只存业务关联元数据

- **前后端直连OSS**：前端直接上传/下载 MinIO，绕过后端服务，提升性能

- **轻量化模拟用户体系**：以自定义用户名作为唯一关联维度，无需登录鉴权，简化演示场景

## 3. 核心业务流程

### 3.1 整体访问链路

用户浏览器 → Kong 网关 → 路由分发至 FastAPI 服务 → 返回静态页面 / 响应接口数据

### 3.2 文件上传完整流程

1. 用户通过浏览器访问系统首页，切换至**上传态页面**

2. 用户输入自定义用户名、选择本地文件

3. 前端页面**直接调用 MinIO 接口**完成文件上传

4. MinIO 接收文件，存储二进制数据，返回文件 Key、访问地址、文件基础信息

5. 前端上传成功后，将「用户名+文件信息+MinIO地址」提交至 FastAPI 后端接口

6. FastAPI 接收数据，校验参数后写入 MongoDB 存储用户文件关联关系

7. 前端收到成功响应，提示文件上传绑定完成

### 3.3 文件下载完整流程

1. 用户切换至**下载态页面**，输入已上传文件的用户名

2. 前端调用 FastAPI 列表查询接口，传入用户名

3. FastAPI 查询 MongoDB，获取该用户绑定的所有文件元数据列表（文件名、MinIO下载地址、文件大小等）

4. 前端渲染文件列表，展示所有关联文件

5. 用户点击下载按钮，**前端直连 MinIO 地址**完成文件下载，无需后端中转

## 4. 页面功能需求

系统包含统一静态页面，支持**上传态/下载态**切换，无需多页面跳转，轻量化实现。

### 4.1 通用功能

- 页面模式切换开关：上传模式 / 下载模式

- 用户名输入框（唯一业务关联字段，非空校验）

### 4.2 上传模式页面功能

- 用户名称输入：自定义任意用户名，用于绑定本次上传文件

- 文件上传入口：支持本地文件选择、批量单文件上传（基础版支持单文件）

- 直传 MinIO：前端直接调用 MinIO S3 接口上传文件

- 上传状态提示：上传中、上传成功、上传失败提示

- 元数据提交：上传成功后自动向后端提交关联数据，完成用户与文件绑定

### 4.3 下载模式页面功能

- 用户名称输入：输入已上传文件的绑定用户名

- 查询按钮：点击后请求后端接口，加载对应用户文件列表

- 文件列表渲染：展示文件名、文件大小、上传时间、下载按钮

- 直连下载：点击下载按钮，浏览器直接请求 MinIO 地址下载文件

- 空数据提示：该用户无绑定文件时展示空状态文案

## 5. 后端接口需求（FastAPI）

后端提供 2 个核心业务接口 + 1 个可选健康检查接口，所有接口统一由 Kong 网关路由转发。

### 5.1 接口通用规范

- 请求协议：HTTP/JSON

- 统一响应格式：`{code:200/500, msg:"提示信息", data:返回数据}`

- 所有接口无登录鉴权，基于用户名做业务关联

### 5.2 文件元数据存储接口（核心）

**接口路径**：/api/file/bind

**请求方式**：POST

**接口描述**：接收前端上传成功后的文件信息与用户名，存储用户-文件关联元数据至MongoDB

**请求参数（Body JSON）**：

- username：string（必填）- 绑定的用户名

- file_name：string（必填）- 原始文件名

- file_key：string（必填）- MinIO 文件唯一Key

- file_url：string（必填）- MinIO 文件访问/下载地址

- file_size：number（必填）- 文件大小（单位：字节）

- mime_type：string（选填）- 文件类型

**响应数据**：返回存储成功的文件唯一ID、绑定用户名

### 5.3 用户文件列表查询接口（核心）

**接口路径**：/api/file/list

**请求方式**：GET

**接口描述**：根据用户名查询该用户绑定的所有文件元数据列表

**请求参数（Query）**：

- username：string（必填）- 待查询的用户名

**响应数据**：返回该用户的文件列表，包含文件名、文件地址、大小、上传时间、文件Key等

### 5.4 健康检查接口（可选）

**接口路径**：/health

**请求方式**：GET

**接口描述**：服务存活检测，用于容器健康检查、网关探活

**响应内容**：`{"status":"ok"}`

## 6. 数据存储设计

### 6.1 MinIO 存储规则

- 单独创建业务Bucket存储用户上传文件

- 存储内容：文件完整二进制数据

- 文件唯一标识：自定义 file_key（避免重名覆盖）

- 访问方式：前端直连 MinIO S3 接口上传、下载

### 6.2 MongoDB 集合设计（file_user_relation）

集合名称：file_user_relation，用于存储用户与文件的关联元数据，核心字段如下：

- \_id：数据库自动生成唯一ID

- username：绑定用户名（建立索引，提升查询速度）

- file_name：原始文件名

- file_key：MinIO 文件唯一Key

- file_url：MinIO 文件下载地址

- file_size：文件大小

- mime_type：文件类型

- create_time：数据创建时间（文件绑定时间）

## 7. Kong 网关路由规则

所有外部请求统一经过 Kong 网关，路由分发规则如下：

1. **静态页面路由**：所有根路径请求、静态资源请求，转发至 FastAPI 静态文件服务

2. **业务接口路由**：/api/ 开头的所有请求，转发至 FastAPI 后端接口服务

3. **健康检查路由**：/health 请求转发至 FastAPI 健康接口

4. **MinIO 路由**：前端直接访问 MinIO 容器端口/域名，无需经过Kong（规避大文件流量占用网关资源）

## 8. 非功能需求

- **轻量化部署**：基于 Docker Compose 一键启动、一键停止、数据持久化

- **数据持久化**：MongoDB、MinIO 均挂载本地数据卷，容器重启数据不丢失

- **高可用适配**：架构支持后续扩容，可升级为集群部署

- **低耦合**：各服务职责单一，可单独更新、重启、替换

## 9. 部署架构总结

本系统采用「Kong网关 + FastAPI业务服务 + MongoDB元数据存储 + MinIO对象存储」的轻量化标准架构，完美适配你的业务诉求：前端直传直下OSS减轻后端压力、数据库仅管理业务关联关系、网关统一流量管控、服务完全容器化，是小型文件托管系统的最优轻量化落地方案。
