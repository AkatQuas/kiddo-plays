# Web端多人视频聊天系统MVP产品需求文档（PRD）

## 1. 产品概述

### 1.1 产品定位

基于 **原生HTML+JavaScript+Tailwind CSS + Express + Socket.io\*\*** + Mediasoup\*\* 搭建的服务端中转（SFU）多人实时音视频聊天系统，为轻量化 MVP 版本，核心解决 Web 端多人实时视频通话需求。摒弃 WebRTC 原生 P2P 带宽瓶颈，通过服务端统一转发音视频流，支持最多4人同房间在线通话。

### 1.2 MVP 核心目标

- 实现用户自主创建视频房间、加入房间能力，单房间最大4人并发

- 基于 Mediasoup SFU 服务中转音视频流，替代 P2P 直连，保证通话稳定性

- Web 端区分展示本地、远端实时音视频画面，状态清晰可辨

- 基础通话能力：摄像头/麦克风开关、退出房间、自动断连回收资源

- 全双工实时通话，支持多人同时发言、视频传输

### 1.3 版本约束（MVP 范围）

- 单房间最大在线人数：**4人（含创建者）**，满员后禁止新用户加入

- 仅支持 Web 端访问，无客户端、小程序适配

- 无用户登录注册，匿名临时入会，房间号为唯一会话标识

- 无录屏、美颜、屏幕共享、聊天文字等增值功能，仅保留音视频通话核心能力

## 2. 整体技术架构设计

### 2.1 架构模式

采用**Node.js + 内嵌 Mediasoup SFU 一体化架构**，无需独立部署第三方媒体服务。Mediasoup 以 NPM 模块形式直接内嵌在 Node 业务进程中，实现「信令控制 + 媒体流中转」单服务闭环，摒弃 WebRTC P2P 带宽瓶颈，所有音视频数据由本地服务统一转发，架构轻量化、部署极简，完全适配4人小房间MVP场景。

### 2.2 架构分层明细

#### 第一层：前端 Web 客户端（用户层）

- 技术栈：原生 HTML + 原生 JavaScript + Tailwind CSS（无Vue/React等框架）

- 核心能力：创建/加入房间、本地媒体采集、音视频轨道推送、远端流渲染、状态展示、WebSocket 信令交互

- 页面核心模块：房间操作区、本地视频窗口、远端视频网格窗口、设备控制按钮区

#### 第二层：Node.js 信令服务（控制层）

- 技术栈：Node.js + Express + Socket.io

- 核心能力：房间创建/销毁、用户进出房间管理、人数校验、转发 SDP/ICE 信令、同步用户在线状态、资源调度通知

- 核心规则：维护全局房间列表，实时校验房间人数，4人满员拦截新入会请求

#### 第三层：内嵌 Mediasoup SFU 媒体能力

- 集成方式：通过 npm 直接内嵌至 Node 服务，**无独立进程、无第三方外部服务依赖**

- 核心能力：运行内部 Worker 媒体进程，创建媒体路由、接收客户端上行音视频流、房间内多点转发、RTP/RTCP 调度、带宽适配、自动资源回收

- 核心优势：单服务完成信令+媒体中转，部署极简、链路短、延迟低，完美适配单房间4人并发的轻量化场景

#### 第四层：基础依赖层

- STUN 服务：公网环境 NAT 穿透，保证内外网设备可正常建联

- HTTPS/WSS 协议：满足浏览器 WebRTC 安全策略要求

### 2.3 核心数据流（全双工中转流程）

客户端媒体流 **上行：客户端 → 内嵌 Mediasoup SFU**，**下行：内嵌 Mediasoup SFU → 其他客户端**。Node 主进程同时承担信令调度与媒体服务管控，信令指令、媒体资源调度均在同一服务内完成，无跨服务网络开销。

1. 用户创建/加入房间，与 Node 信令服务建立 Socket 长连接

2. 信令服务通知 Mediasoup 创建对应房间的媒体路由、消费者/生产者通道

3. 客户端采集本地音视频，生成媒体轨道，推流至 Mediasoup 服务（生产者 Producer）

4. Mediasoup 接收上行流，同步转发给房间内所有其他在线用户（消费者 Consumer）

5. 客户端拉取服务端转发的远端流，渲染至页面窗口，实现双向实时通话

## 3. 核心功能需求

### 3.1 房间管理功能

- **创建房间**：用户点击创建房间，系统自动生成唯一随机房间号（6位数字），创建成功后自动进入房间，开启本地媒体采集

- **加入房间**：用户输入有效房间号，校验房间存在且人数＜4人，即可加入房间；房间满4人时，提示「房间人数已满，无法加入」

- **房间销毁**：房间内所有用户退出后，Node 服务自动销毁房间，Mediasoup 回收对应媒体资源，释放带宽与 CPU

- **人数限制**：单房间最大4人，实时展示当前在线人数/最大人数（如 3/4）

### 3.2 音视频通话功能

- **本地媒体展示**：页面固定窗口展示本地实时视频、麦克风收音状态，明确标注「自己」，与远端画面区分

- **远端媒体展示**：采用网格布局自适应展示1-3个远端用户画面，每个远端画面独立标注用户标识，状态独立

- **设备控制**：支持一键关闭/开启摄像头、关闭/开启麦克风，关闭后页面实时展示静默/黑屏状态

- **全双工通话**：支持多人同时说话、传输视频，无单向阻塞，兼容正常对话场景

- **自动重连**：短暂网络波动时，自动重试建联；断连超时后自动退出房间并提示

### 3.3 状态展示功能

- 房间基础信息：展示当前房间号、在线人数（当前/最大）

- 媒体状态：本地/远端摄像头开启/关闭、麦克风开启/关闭实时状态

- 连接状态：通话中、建联中、断连、房间满员等状态提示

### 3.4 退出功能

- 用户主动点击退出房间：关闭本地媒体轨道、断开 Socket 连接、释放页面资源

- 关闭页面/浏览器：后台自动检测离线，10s内回收用户资源，更新房间人数

## 4. 页面 UI 设计规范（MVP 极简版）

### 4.1 首页（房间操作页）

- 核心模块：创建房间按钮、输入框+加入房间按钮、操作提示文案

- 交互：点击创建房间 → 跳转通话页；输入房间号点击加入 → 校验通过跳转通话页

### 4.2 通话页面（核心页面）

- **布局规则**：本地视频窗口固定悬浮/置顶，远端视频采用自适应网格（1-3个画面自动适配排版）

- **画面区分规则**：本地窗口标注「我」，边框高亮区分；每个远端窗口标注「用户1/用户2/用户3」

- 底部控制栏：麦克风开关、摄像头开关、退出房间按钮

- 顶部状态栏：房间号、当前在线人数

## 5. 完整业务使用流程

### 5.1 流程1：用户A创建房间

1. 用户A打开Web页面，点击【创建房间】按钮

2. 前端请求Node信令服务，服务生成6位随机房间号，创建空房间

3. 信令服务调用Mediasoup，初始化该房间的媒体路由、生产者/消费者通道

4. 前端申请浏览器摄像头、麦克风权限，采集本地媒体流

5. 本地流推送至Mediasoup服务，页面渲染本地视频画面，房间状态更新为1/4人在线

6. 页面展示当前房间号，可复制分享给其他用户

### 5.2 流程2：用户B/C/D加入房间

1. 用户B打开页面，输入用户A分享的房间号，点击【加入房间】

2. 前端请求Node服务，校验房间存在、当前人数＜4，校验通过允许入会

3. 与Node信令、Mediasoup媒体服务建立双连接，申请本地媒体权限

4. 本地音视频流推送到Mediasoup服务

5. Mediasoup将用户B的流转发给房间内所有在线用户（A），同时将A的流转发给B

6. 双方页面同步渲染本地+远端画面，实现双向通话，房间人数更新为2/4

7. C/D用户重复上述流程，直至房间满4人，后续用户无法加入

### 5.3 流程3：用户退出房间

1. 用户点击【退出房间】或关闭浏览器页面

2. 前端停止本地媒体采集，关闭音视频轨道，断开Socket连接

3. Node服务更新房间在线人数，通知Mediasoup销毁该用户的媒体通道

4. 房间内其他用户页面自动移除该用户的远端视频窗口，更新人数状态

5. 房间最后一人退出后，服务自动销毁房间，释放所有媒体资源

## 6. 技术核心流程（底层交互）

1. **信令建联**：前端页面初始化，与Node Socket服务建立长连接，注册用户会话

2. **房间初始化**：创建/加入房间后，信令服务分配Mediasoup Worker、Router媒体路由

3. **生产者创建**：客户端采集媒体流，与SFU协商SDP，创建Producer上传上行流

4. **消费者订阅**：检测到房间内新流，自动创建Consumer，订阅服务端转发的远端流

5. **实时中转**：SFU持续转发各路音视频流，维持全双工通话

6. **资源回收**：用户退出/断连，逐级销毁Consumer、Producer、媒体路由，释放资源

## 7. 非功能需求（MVP 基础标准）

- **并发能力**：单房间稳定支持4人实时通话，无卡顿、音画不同步问题

- **延迟标准**：公网环境音视频延迟 ≤ 300ms，满足日常通话需求

- **兼容性**：支持 Chrome、Edge、Firefox 最新版浏览器

- **稳定性**：单房间持续通话30分钟无闪退、无断流、无内存泄漏

- **技术规范性**：前端纯原生语法+Tailwind样式统一规范，服务端Express轻量化架构，代码简洁易部署、易维护

## 8. 技术栈最终定型 & MVP 功能目标

### 8.1 最终定型技术栈

- **前端技术栈**：原生HTML、原生JavaScript、Tailwind CSS（零框架、轻量化、浏览器零依赖）

- **服务端技术栈**：Node.js + Express（服务基座）+ Socket.io（信令通信）+ Mediasoup（内嵌SFU媒体中转）

- **通信协议**：WSS加密信令（可选）、WebRTC媒体传输、RTP/RTCP流调度

### 8.2 MVP 必做功能目标

- 房间创建、6位房间号生成、房间号分享、加入房间、4人满员拦截限制

- Express+Socket.io信令服务 + 内嵌Mediasoup媒体中转完整链路打通

- 前端原生实现本地/远端音视频采集、渲染，Tailwind自适应布局区分本地、远端画面

- 实时设备控制：麦克风/摄像头一键开关、状态实时展示

- 用户主动退出、页面异常关闭自动断连、服务端自动回收媒体与房间资源

- 网络波动自动重连、房间人数/连接状态实时提示

在项目 ./web-meeting 实现上述项目的所有功能代码，MVP 。
