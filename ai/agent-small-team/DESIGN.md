# Agent Team Mode MVP 产品需求文档

## 一、产品概述

### 1.1 产品定位

本产品是基于 @openai/agents 底座构建的**多智能体团队协作执行系统**，通过「规划-执行-审核-汇总」的专业化角色分工，形成完整任务闭环，将复杂用户需求拆解为原子任务并行执行，经校验后输出标准化结果。
MVP 阶段以 Node.js 命令行形态交付，无前端交互界面，核心验证多智能体团队协作模式的流程可行性、角色分工合理性与全链路可观测能力。

### 1.2 核心价值

1. **专业分工提效**：不同智能体专注单一环节，相比单智能体全链路执行，任务拆解更精准、执行更聚焦、审核更客观
2. **结果质量可控**：内置审核-重试闭环，对不合格结果自动打回重执行，保障最终输出符合验收标准
3. **全流程可回溯**：全链路埋点追踪日志，支持问题定位、效果复盘与智能体调优
4. **并行执行提速**：无依赖的子任务支持多 Worker 并行处理，缩短整体任务耗时

### 1.3 目标用户

MVP 阶段核心面向 AI 应用开发者、智能体系统调试人员，用于验证多智能体协作模式、执行复杂拆解类任务、调试智能体角色 Prompt 与流程逻辑。

## 二、产品目标与范围

### 2.1 MVP 核心目标

1. 跑通「Planner 规划 → Worker 并行执行 → Reviewer 审核打回 → Reporter 汇总」的完整团队协作闭环
2. 落地全局共享上下文 + 角色私有上下文的双层上下文管理机制
3. 实现全流程每一步的追踪日志埋点，支持完整链路回溯与问题排查
4. 验证基于 @openai/agents 底座的多实例智能体调度可行性

### 2.2 需求范围

#### 包含范围

- 5类核心智能体角色的完整职责与执行逻辑
- 团队调度中枢的主循环控制、状态流转、重试管理
- 全局工具池统一管理、各智能体按需调用
- 双层上下文（全局共享+角色私有）的数据隔离与共享规则
- 全链路追踪日志的埋点、存储与控制台输出
- 命令行一次性输入、一次性输出的交互模式
- 不合格任务自动打回重试机制

#### 不包含范围

- 可视化前端管理界面、交互控制台
- 日志持久化存储（仅内存留存，控制台打印）
- 动态工具注册、工具权限分级管理
- 多轮对话上下文持久化、历史会话管理
- Worker 任务优先级调度、限流与队列管理
- 智能体角色自定义配置、动态新增角色
- 失败任务降级输出、部分结果兼容展示

## 三、核心角色定义

### 3.1 系统外角色

| 角色              | 角色描述                           | 核心诉求                                           |
| ----------------- | ---------------------------------- | -------------------------------------------------- |
| 任务提交者        | 通过命令行输入任务需求的使用者     | 输入需求后自动得到结构化最终结果，无需干预中间过程 |
| 系统研发/调试人员 | 负责优化智能体逻辑与流程的开发人员 | 可查看全链路执行日志，定位问题环节，调优各角色能力 |

### 3.2 系统内智能体角色

所有智能体基于 @openai/agents 实例化，共享全局上下文与全局工具池，同时各自持有独立私有上下文。

#### 1. Planner（任务规划师）

**核心职责**

- 接收用户原始需求，拆解为多个可独立执行的原子子任务
- 识别子任务间的依赖关系，标注可并行执行的任务
- 为每个子任务明确所需调用的工具范围
- 为每个子任务制定清晰的验收标准，作为后续审核依据
- 输出标准化结构化任务清单，供调度中枢分发执行

**能力边界**

- 仅负责任务拆解与规划，不执行具体任务，不调用工具
- 不对最终结果质量负责，仅保障拆解逻辑的合理性

**私有上下文内容**

- 角色固定行为准则与输出规范
- 任务拆解思路备注（用于回溯优化）
- 已生成的任务清单缓存

#### 2. Worker（任务执行员，多实例）

**核心职责**

- 接收调度中枢分配的单条子任务
- 依据任务要求调用全局工具池中的对应工具完成执行
- 完整记录工具返回的原始数据与执行过程
- 输出当前子任务的原始执行结果

**能力边界**

- 单实例仅负责单条子任务，不跨任务处理
- 不对结果做整合、校验与润色，仅输出原始执行产出
- 多实例之间相互隔离，无直接数据交互

**私有上下文内容**

- 角色固定行为准则与输出规范
- 当前分配的子任务详情（任务ID、任务内容、所需工具）
- 对应子任务的验收标准
- 执行过程中间结果与原始工具返回数据

#### 3. Reviewer（结果审核员）

**核心职责**

- 批量接收所有 Worker 的执行结果，对照 Planner 预设的验收标准逐条校验
- 校验维度：内容完整性、数据准确性、工具调用合规性、需求匹配度
- 将结果分为三类：审核通过、存在缺陷需重执行、完全无法执行
- 为不合格任务标注详细打回理由与修正要求，供重执行参考

**能力边界**

- 仅做结果校验，不修改执行结果
- 不参与任务拆解与重新规划，仅针对已有任务的执行质量判定

**私有上下文内容**

- 角色固定行为准则与审核标准
- 全部 Worker 执行结果缓存
- 审核通过的任务结果清单
- 不合格任务清单与打回原因

#### 4. Reporter（报告汇总员）

**核心职责**

- 接收所有审核通过的子任务结果
- 贴合用户原始需求的逻辑，整合、润色、结构化输出最终报告
- 补充任务执行总览摘要、关键数据汇总
- 同时输出自然语言总结与结构化明细两种形式

**能力边界**

- 仅整合已审核通过的结果，不补充额外数据
- 不对原始结果做事实性修改，仅做逻辑组织与表达优化

**私有上下文内容**

- 角色固定行为准则与输出规范
- 全部合格子任务结果数据
- 生成的最终报告缓存

#### 5. Team Orchestrator（团队调度中枢）

非 LLM 智能体，为系统主控逻辑层，负责全流程驱动与资源管理。
**核心职责**

- 接收用户输入，初始化全局上下文与追踪日志体系
- 驱动全流程状态流转，按顺序调用各智能体角色
- 管理 Worker 多实例并行执行，收集执行结果
- 管控重试逻辑：统计重试次数、下发重执行任务、终止超限任务
- 统一埋点全链路追踪日志，管理日志存储与输出
- 统一异常捕获与处理，保障流程可控

## 四、核心业务主流程

### 4.1 流程总览

整体为**单轮闭环+内部重试循环**结构：一次用户输入触发一次完整团队协作流程，执行-审核环节内部支持多次重试，直至全部任务通过审核或达到重试上限。

### 4.2 分步流程详解

1. **任务输入与初始化**
   - 用户通过命令行输入原始任务需求
   - 调度中枢生成全局唯一链路 ID，初始化全局上下文（含用户输入、全局工具池、全局配置）与追踪日志存储
   - 记录首条追踪日志：任务初始化完成

2. **任务规划阶段**
   - 调度中枢调用 Planner 智能体，传入全局上下文与用户原始需求
   - Planner 完成任务拆解，输出结构化任务清单（含任务ID、内容、所需工具、并行属性、验收标准）
   - 调度中枢记录规划阶段追踪日志，留存输入输出与耗时

3. **并行执行阶段**
   - 调度中枢根据任务清单，为每个子任务实例化独立的 Worker 智能体
   - 可并行的子任务同步启动执行，各 Worker 加载自身私有上下文与全局工具
   - 所有 Worker 执行完成后，调度中枢统一收集全部子任务执行结果
   - 记录批量执行与单任务两层追踪日志

4. **结果审核阶段**
   - 调度中枢调用 Reviewer 智能体，传入任务清单与全部 Worker 执行结果
   - Reviewer 对照验收标准逐条审核，输出审核通过清单与不合格清单（含打回理由）
   - 调度中枢记录审核阶段追踪日志

5. **重试判定与循环**
   - 若不存在不合格任务，直接进入汇总阶段
   - 若存在不合格任务，调度中枢校验单任务已重试次数：
     - 未达最大重试上限：标记重试次数，将任务重新下发对应 Worker 执行，回到第3步，同时记录重试日志
     - 达到最大重试上限：终止流程，抛出任务执行失败异常，记录异常日志

6. **报告汇总阶段**
   - 全部任务审核通过后，调度中枢调用 Reporter 智能体，传入所有合格结果
   - Reporter 整合内容，生成最终结构化报告
   - 调度中枢记录汇总阶段追踪日志

7. **结果输出与链路回溯**
   - 控制台输出最终汇报结果
   - 控制台按时序输出全链路追踪日志，展示各阶段耗时、状态、输入输出摘要
   - 流程结束，释放上下文资源

## 五、核心功能需求

### 5.1 团队调度中枢

1. **全流程状态管理**：维护任务全局状态（初始化/规划中/执行中/审核中/重试中/汇总中/完成/失败），状态流转严格按流程执行，不可跳变
2. **并行调度能力**：支持多个 Worker 实例同时执行，自动等待全部任务完成后再进入下一环节
3. **重试管控能力**：支持配置全局最大重试次数，单任务独立统计重试次数，超限强制终止
4. **统一异常处理**：捕获 LLM 调用异常、工具执行异常、流程超时异常，记录异常信息到追踪日志，友好输出错误原因

### 5.2 任务规划能力

1. **原子化拆解**：将复杂需求拆分为无重叠、可独立执行的最小任务单元，避免任务交叉依赖
2. **依赖识别**：准确识别任务间的先后依赖关系，标注可并行与必须串行的任务
3. **验收标准制定**：每个子任务配套可量化、可校验的验收标准，无模糊描述
4. **结构化输出**：统一输出标准格式的任务清单，无多余自然语言，保障调度中枢可直接解析

### 5.3 并行任务执行能力

1. **多实例隔离**：每个子任务对应独立 Worker 实例，上下文与执行过程完全隔离
2. **工具按需调用**：Worker 仅可调用任务指定范围内的全局工具，不可越权使用
3. **原始结果留存**：完整保留工具返回的原始数据，不做擅自删减与修改
4. **异常上报**：执行过程中出现工具调用失败、数据缺失等问题，需清晰标注异常原因，随结果一并返回

### 5.4 结果审核与重试能力

1. **批量审核**：支持一次性审核所有子任务结果，效率与单任务审核一致
2. **多维度校验**：覆盖完整性、准确性、合规性、需求匹配度四个核心维度
3. **缺陷可追溯**：每个不合格任务必须附带明确的打回理由与修正方向，不可笼统判定不合格
4. **重试上下文继承**：重执行时保留上一轮执行结果与打回意见，供 Worker 参考修正

### 5.5 报告汇总输出能力

1. **逻辑整合**：按用户需求的业务逻辑组织内容，而非简单罗列子任务结果
2. **双形态输出**：同时提供自然语言总结（便于阅读）与结构化明细（便于程序解析）
3. **冗余过滤**：去除中间执行日志、工具调用细节等冗余信息，仅保留有效结论
4. **总览补充**：增加任务执行摘要，说明任务拆解数量、执行情况、整体结论

### 5.6 双层上下文管理机制

1. **全局共享上下文**
   - 所有智能体均可读取，不可私自修改
   - 包含内容：用户原始输入、全局工具池定义、全局配置参数（最大重试次数、超时时间）、全链路追踪日志、任务全局状态
2. **角色私有上下文**
   - 各智能体独立持有，其他角色不可访问
   - 包含内容：角色固定行为指令、当前任务专属参数、角色内中间结果缓存、角色专属配置
3. **上下文传递规则**：调度中枢负责向各智能体注入全局上下文与对应私有上下文，智能体执行完成后仅返回产出结果，不修改全局上下文

### 5.7 全链路追踪与回溯能力

1. **全节点埋点**：规划、批量执行、单任务执行、审核、汇总五个核心环节必须埋点，无遗漏
2. **日志字段规范**：每条追踪日志必须包含：全局链路ID、步骤名称、执行角色、时间戳、执行耗时、入参摘要、出参摘要、状态标记（正常/重试/异常）、异常信息（如有）、重试次数（如有）
3. **长内容截断**：入参出参摘要支持超长内容自动截断，保留核心信息，保障日志可读性
4. **时序化输出**：流程结束后自动按时间升序输出完整链路日志，标注各阶段相对启动时间的耗时
5. **重试标记**：重执行的任务日志需明确标记重试次数与上一轮失败原因，便于区分多次执行

## 六、非功能需求

### 6.1 可观测性

- 核心流程步骤日志覆盖率 100%，关键异常必须有明确日志记录
- 链路日志可完整还原单次任务的全执行过程，支持定位问题发生的具体环节与原因

### 6.2 可靠性

- 单 Worker 执行失败不影响其他 Worker 正常运行，不导致整体流程崩溃
- 重试机制可覆盖大部分执行质量问题，最大重试次数内通过率不低于 80%
- 异常场景均有友好提示，不会出现无响应、无日志的静默失败

### 6.3 扩展性

- 智能体角色支持后续新增，无需重构主流程框架
- 全局工具池支持后续扩展工具，无需修改调度逻辑
- 追踪日志体系支持后续对接持久化存储与可视化平台

### 6.4 性能

- 可并行任务必须并行执行，整体耗时不超过最长单任务耗时的 1.2 倍
- 单轮流程（不含重试）内，调度与日志处理的额外耗时占比不超过 10%

## 七、MVP 验收标准

1. 可通过命令行一次性输入任务需求，自动跑完「规划-执行-审核-汇总」完整闭环
2. 支持至少 3 个 Worker 实例并行执行独立子任务
3. 对审核不合格的任务可自动打回重执行，支持最多 2 次重试，超限后明确报错
4. 全流程 5 个核心环节均有追踪日志，流程结束后可在控制台输出完整时序链路
5. 各智能体私有上下文相互隔离，全局工具与基础上下文统一共享
6. 所有智能体实例均基于 @openai/agents 底座构建
7. 最终输出包含自然语言总结与结构化明细两部分内容
