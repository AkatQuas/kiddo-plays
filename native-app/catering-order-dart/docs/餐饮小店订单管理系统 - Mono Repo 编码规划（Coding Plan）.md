# 餐饮小店订单管理系统 - Mono Repo 编码规划（整合Makefile统一入口版）
## 一、Mono Repo 整体架构规划
### 1.1 核心设计原则
- 统一Dart技术栈，抽离公共模块最大化复用，降低单人维护成本；
- 严格遵循「数据/网络/部署/权限」四层隔离，双应用代码物理分离、逻辑解耦；
- 适配纯本地部署特性，无云端依赖，编译产物轻量化；
- 工程化配置统一，新增**顶层Makefile作为全局唯一操作入口**，封装所有脚本调用，简化开发与构建流程；
- 版本/依赖/构建/清理全流程标准化，错误提前拦截，自带自助式帮助文档。

### 1.2 调整后Mono Repo目录结构
```
catering-order-system/          # 根目录
├── Makefile                    # 新增：顶层统一构建入口（跨平台兼容，Windows优先）
├── packages/                   # 公共复用模块（核心抽离）
│   ├── common_core/            # 基础核心包（全项目复用）
│   ├── database_core/          # 数据库核心包（Sqlite+Drift统一封装）
│   ├── offline_voice/          # 离线语音能力包（TTS+ASR封装）
│   └── auth_core/              # 权限核心包（双模式鉴权、超时管控）
├── apps/                       # 双应用工程目录（物理隔离）
│   ├── merchant_app/           # 商家端Android APP（Flutter Android）
│   └── ops_tool/               # 私有化运维管理工具（前后端分离）
├── scripts/                    # 底层执行脚本（仅被Makefile调用，不直接手动执行）
│   ├── outputs/                # 新增：统一构建产物归档目录（按产物类型分子目录）

├── pubspec.yaml                # 根依赖（统一管理版本、依赖约束）
└── README.md                   # 项目说明（部署、迭代、隔离规则）
```

## 二、工程化基础配置（Phase 1）
### 2.1 核心配置项
1. **依赖管理**：根目录`pubspec.yaml`声明公共包版本约束，子应用通过`path`依赖`packages/`下模块，确保版本统一；
2. **编译约束**：
   - 商家端：禁用网络权限（AndroidManifest.xml）、配置离线打包参数，禁止热更新；
   - 运维服务端：配置Dart编译为Windows独立exe（无运行时依赖）；
   - 运维前端：编译为纯静态Web资源（无跨域、无外部CDN）；
3. **版本管控**：全局维护`version_constants.dart`，统一数据库版本号、应用版本号，对接「版本锁」拦截降级逻辑；
4. **Makefile入口配置**：
   - 设计为跨平台兼容，自动识别Windows/类Unix系统，适配不同脚本执行方式；
   - 采用伪目标声明，避免与同名文件冲突；
   - 默认目标为帮助文档，自动解析所有命令及说明，无需额外查阅文档。

### 2.2 初始化任务

- 配置Flutter Android/Flutter Web/Dart Server多工程编译环境；
- 编写基础工程化脚本（依赖安装、代码规范检查）；
- 完成顶层Makefile框架搭建，封装环境检查、依赖安装、代码检查三类基础命令。

## 七、统一构建与部署体系（Phase 5，原“构建与部署脚本开发”重构）
### 7.1 顶层Makefile核心能力规划
按功能模块划分所有命令，实现“一条命令完成对应全流程”，无需手动切换目录或执行多个脚本：
1. **环境与依赖模块**：封装开发环境检查、全项目依赖安装、全项目代码规范检查命令，自动遍历所有packages和apps目录执行对应操作；
2. **构建产物模块**：分别封装商家端APK、运维前端静态资源、运维服务端EXE的构建命令，新增一键构建所有支持产物的命令（Windows自动构建全部，类Unix系统自动跳过运维服务端）；
3. **版本管理模块**：封装版本号升级（支持patch/minor/major三种类型）、版本一致性检查命令，自动同步应用版本与数据库版本；
4. **清理模块**：封装全项目清理命令，自动删除所有构建产物、临时文件和依赖缓存，确保构建环境干净。

### 7.2 底层脚本标准化改造
所有`scripts/`下的脚本仅作为底层执行单元，统一规范输入输出，由Makefile统一调用：
1. **统一输出目录**：所有构建产物不再分散在各应用的build目录，构建完成后自动复制到`scripts/outputs/`下对应子目录，并按版本号命名归档；
2. **错误处理标准化**：所有脚本开启错误立即退出模式，执行失败时返回明确错误信息，由Makefile统一捕获并提示；
3. **平台适配**：Windows批处理脚本与类Unix Shell脚本功能对齐，确保Makefile调用时无需区分平台。

### 7.3 标准使用流程
1. 首次开发：执行`make doctor`检查环境 → 执行`make install`安装所有依赖；
2. 日常开发：执行`make lint`检查代码规范；
3. 构建交付：执行对应构建命令（如`make build-merchant`），产物自动归档到统一目录；
4. 版本迭代：执行`make version-bump TYPE=xxx`升级版本 → 执行`make build-all`构建所有产物；
5. 环境重置：执行`make clean`清理所有临时文件。
